Javelin is a real-time rigid body physics engine written from scratch in **C++23**, using the language's new module system throughout. It handles spheres and oriented boxes, resolves contact with a projected Gauss-Seidel solver, persists contact manifolds across frames for warm starting, and parallelizes its broad phase across all hardware threads. A custom OpenGL renderer visualizes the simulation, and a text-based scene format lets scenes be authored and inspected without tooling.

### Architecture

The engine is organized around a fixed-timestep loop running on a dedicated `std::jthread`. The render thread and the physics thread share state through a lock-free **pose channel** — the physics thread writes authoritative world transforms once per tick; the render thread reads the latest snapshot without blocking.

Each physics tick follows a strict five-stage pipeline:

1. **Force integration** — gravity applied to all dynamic bodies; per-body AABB bounds updated for broad phase.
2. **Broad phase** — candidate collision pairs generated by querying a dual-BVH structure across a worker thread pool.
3. **Narrow phase** — exact contact manifolds computed for each candidate pair, with stable feature IDs for cross-frame matching.
4. **Solve** — iterative projected Gauss-Seidel in velocity space, followed by a positional correction pass for residual penetration.
5. **Publish** — authoritative world transforms written to the pose channel for the renderer.

---

### Broad Phase

The broad phase uses two separate BVH structures: a **dynamic BVH** that is updated incrementally each tick for all moving bodies, and a **static BVH** that is rebuilt only when the body count changes. Leaf bounds are fattened so small motions don't require a remove/insert cycle; a `contains` check avoids rebuilding the tree when the fat bounds still enclose the new tight bounds.

Broad phase queries are parallelized across all hardware threads using a **persistent worker pool** — threads sleep on a condition variable and wake when the main physics thread posts a new job. Each worker owns a contiguous chunk of dynamic body IDs and queries the BVH independently with its own scratch buffers. Results are merged in deterministic order, canonicalized (min-id first), and deduplicated before being handed to the narrow phase.

A separate microbenchmark driver stress-tests the dynamic BVH in isolation: 20,000 bodies, 2,000 queries per iteration, configurable world extent and body radius.

### Narrow Phase

For each candidate pair from the broad phase, the narrow phase generates a **contact manifold** — a contact normal, up to four contact points, penetration depth, and per-point local anchor offsets in each body's frame.

Supported shape pairs:
- **Sphere–sphere**: direct closest-point test.
- **Sphere–box**: GJK-style support point query against the box, followed by contact normal derivation.
- **Box–box**: Separating Axis Theorem with axis hysteresis to prevent axis flipping between frames. Face-face contacts produce up to four clipped contact points via polygon intersection; edge-edge contacts produce one.
- **Box–ground** / **Sphere–ground**: implicit infinite ground plane.

Every contact point carries a **feature ID** — a compact bitfield encoding the contact topology (face index, edge pair, or vertex). These IDs are the key to the persistence system.

### Manifold Persistence and Warm Starting

A naive solver discards all contact information between frames and starts from scratch each tick. This causes visible jitter in stacked objects because the solver needs many iterations to rebuild consistent impulse state.

Javelin persists manifolds across frames. At the start of each tick, contact points in the new manifolds are matched against their counterparts from the previous frame. Matching uses feature IDs first (a point with the same edge or face feature is the same contact point), then falls back to minimum local-anchor distance. A matched point inherits the previous frame's accumulated normal and friction impulses, which are applied as a **warm start** at the beginning of the solve.

Points are dropped from the cache when the world-space separation between anchors exceeds thresholds for normal drift or tangential drift — preventing stale impulses from corrupting contacts that have genuinely changed geometry. The system tracks per-tick statistics (match rate, dropped points, axis flip count) and exposes them as Tracy plots.

### Solver

The contact solver runs in two passes.

**Velocity pass (16 iterations, projected Gauss-Seidel):** For each contact point, it computes the effective mass along the contact normal and two friction tangents, applies a penetration bias to push overlapping bodies apart, and iteratively applies clamped normal impulses (non-negative — bodies can't pull) and Coulomb-clamped friction impulses. Accumulated impulses from warm starting are re-projected at the start to account for frame-to-frame normal rotation.

**Position pass (4 iterations):** After velocity correction, residual penetration is addressed with direct position and orientation correction proportional to `inv_mass`. This prevents objects from slowly sinking into each other even when the velocity solver has converged.

Resting bodies — those with low linear and angular speed while in contact — have their velocities zeroed after the solve to prevent jitter accumulation.

### Rendering

The renderer is an OpenGL 4.6 rasterizer with three passes. The **geometry pass** renders all bodies as instanced primitives (sphere or box mesh) with per-instance transform, scale, and material ID packed into vertex attributes and uploaded as a streaming buffer each frame from the latest pose snapshot. The **world grid pass** draws an infinite grid for spatial reference. The **display pass** composites the scene with a simple directional light, per-material colors, and ACES-style tone mapping.

The renderer is deliberately lightweight — its purpose is visualization and debugging, not production rendering. Tracy GPU zones are inserted at each pass.

### C++23 Modules

The entire codebase uses C++23 **named modules** (`export module javelin.physics.solver;`, `import javelin.physics.types;`) rather than headers. This enforces clean separation of interface from implementation, eliminates include-order dependencies, and keeps compile times predictable as the codebase grows. It also means every module boundary is an explicit contract.

The math library (`vec2`, `vec3`, `vec4`, `mat3`, `mat4`, `quat`) is implemented from scratch in the same module system. Quaternion integration uses the derivative form `q' = 0.5 * ω * q` for correct orientation stepping.
